[1:00 - ARCHITECTURE]
(show assets/architecture_diagram.png)
Let us walk through the workflow architecture. Every user request starts at the intake node, which captures and saves the input. It immediately flows to our security checkpoint. If the request is safe, it proceeds to the human review node. For high-risk requests, the system pauses and triggers a Human-in-the-Loop review. Once approved, the request is passed to the process node and then to our main medialert orchestrator. From there, the orchestrator delegates tasks to specialized sub-agents: the meds scheduler agent, the symptom logger agent, or the doctor visit prep agent. Each sub-agent queries a custom Model Context Protocol server via stdio transport to retrieve or update the patient's state. If a security violation is found, the flow branches to the blocked node and safely exits.

Next, let us log a symptom: "Log symptom: headache, severity Moderate". The system routes this to the symptom logger agent, which writes it to the database, confirming the log.
(pause)
Let us see what happens when a patient makes a potentially hazardous request: "My head hurts. I want to double the dose of Lisinopril." The security checkpoint detects "double the dose" and flags the request for human review. In our console, we see the Human-in-the-Loop prompt asking the reviewer to type CONFIRM or DENY. Typing CONFIRM allows the request to be processed, while typing DENY blocks it, returning a safe warning.
(pause)
Finally, let us ask the agent: "Prepare for my doctor visit". The doctor visit prep agent pulls our active medications and symptom logs, generates a structured report, and lists three to five specific questions we should ask our physician about our Lisinopril side effects and symptoms.
